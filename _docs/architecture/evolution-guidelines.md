# Evolution Guidelines

## Цель документа

Этот документ фиксирует правила, по которым проект стоит развивать дальше, чтобы MVP не превратился в набор связанного между собой кода без явных границ ответственности.

## Базовые принципы

### 1. Держать transport отдельно от application logic

Все, что связано с Telegram update objects, filters, router и отправкой сообщений, должно оставаться в `app/bot`.

Не нужно:

- помещать в handlers логику построения prompt
- помещать в handlers работу с моделью
- помещать в handlers сложную бизнес-логику

### 2. Развивать сценарии через сервисы

Если появятся новые функции, orchestration лучше добавлять в отдельные сервисы, а не расширять один `chat_handler`.

Примеры:

- `ConversationService`
- `UserSettingsService`
- `DocumentIngestService`

### 3. Изолировать внешние интеграции

Любой внешний сервис должен иметь собственный client/adapter слой.

Примеры:

- `OpenAIClient`
- `VectorStoreClient`
- `TranscriptionClient`

Это позволит:

- централизовать retry и error mapping
- не размазывать HTTP-логику по проекту
- проще тестировать сервисный слой

### 4. Typed exceptions как стабильный контракт

Если добавляются новые инфраструктурные сбои, их лучше поднимать наверх через предсказуемые исключения, а не через сырой `Exception`.

### 5. Конфигурация должна оставаться централизованной

Новые env-переменные стоит описывать и валидировать в `app/config.py`, а не читать напрямую в произвольных модулях.

## Как безопасно расти

### Если появится память диалога

Тогда стоит выделить:

- отдельный abstraction для хранения истории
- policy принятия решения, какую историю включать в prompt
- слой репозитория или storage adapter

Важно:

- не смешивать storage-логику с Telegram handlers

### Если появится БД

Тогда стоит добавить:

- отдельный модуль доступа к данным
- миграции
- явные схемы данных

### Если появятся несколько LLM-провайдеров

Тогда стоит:

- ввести общий интерфейс model client
- оставить провайдер-специфичные реализации в `clients/`
- держать выбор провайдера в конфиге

### Если появится streaming

Тогда придется пересмотреть:

- контракт между service и handler
- политику отправки промежуточных сообщений в Telegram
- обработку таймаутов и отмены

## Антипаттерны, которых лучше избегать

- толстые handlers на сотни строк
- прямой доступ к env из разных модулей
- смешивание Telegram API и вызова модели в одном месте
- проброс сырого SDK-ответа модели глубоко в приложение
- хранение скрытого состояния в глобальных переменных
- раннее введение сложных абстракций без реального кейса

## Рекомендуемая следующая документация

Когда проект станет сложнее, имеет смысл добавить:

- ADR по крупным архитектурным решениям в `docs/decisions/`
- отдельную спецификацию env и operational runbook
- sequence diagrams для новых сценариев
- правила prompt management
- документ по тестовой стратегии
