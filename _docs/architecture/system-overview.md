# System Overview

## Назначение

Проект представляет собой Telegram-бота, который принимает текстовые сообщения пользователя, отправляет их в локальную LLM через Ollama и возвращает текстовый ответ обратно в Telegram.

Система намеренно простая и ограниченная:

- только текстовые сообщения
- только polling
- без хранения истории диалога
- без базы данных
- без RAG, tools, voice, image

Это MVP для отработки базовой интеграции Telegram + локальная LLM.

## Границы системы

Внутри системы:

- Telegram bot application на Python
- orchestration входящего сообщения
- вызов локального Ollama
- user-facing обработка ошибок
- разбиение длинных ответов под лимиты Telegram

Вне системы:

- Telegram Bot API как внешний transport
- Ollama как внешний локальный runtime модели
- конкретная модель Ollama как внешняя зависимость окружения
- секреты и окружение как внешний input через `.env`

## Архитектурный стиль

Текущая архитектура - небольшой модульный монолит с явным разделением по слоям.

Слои:

1. Interface layer
   Telegram handlers и router
2. Application layer
   `ChatService`, orchestration одного пользовательского запроса
3. Infrastructure layer
   `OllamaClient`, конфиг, логирование
4. Shared utilities
   ошибки, разбиение текста, prompt-файлы

Такой стиль выбран потому, что он:

- прост для MVP
- прозрачен для чтения
- не требует раннего усложнения
- позволяет потом аккуратно добавлять новые сценарии

## Ключевые архитектурные решения

### 1. Сообщения независимы

Каждый входящий текстовый запрос обрабатывается отдельно. Контекст прошлых сообщений не сохраняется ни в памяти, ни во внешнем хранилище.

Причины:

- это часть исходных требований
- так проще контролировать поведение MVP
- проще отлаживать ошибки и взаимодействие с моделью

### 2. Ollama изолирован клиентом

Вся логика обращения к модели вынесена в `app/clients/ollama_client.py`.

Причины:

- handlers не должны знать детали SDK/HTTP
- легче менять реализацию или провайдера
- удобнее централизованно преобразовывать ошибки

### 3. Telegram не содержит бизнес-логики модели

Handlers отвечают за прием апдейта, вызов сервиса и отправку результата. Решение, как подготовить запрос к модели, сосредоточено в сервисе.

### 4. Конфигурация централизована

Все настройки загружаются через `app/config.py` и валидируются на старте.

### 5. Prompt вынесен в файл

System prompt хранится отдельно от кода, что позволяет менять поведение модели без изменения Python-модулей.

## Текущие ограничения

- один бот-процесс
- нет очередей и фоновых задач
- нет персистентности
- нет stream-ответов
- нет rate limiting
- нет метрик и health endpoints
- нет выделенного доменного слоя, потому что доменная логика пока минимальна

## Целевой вектор развития

При развитии проекта стоит сохранять текущий принцип:

- Telegram-слой не знает детали модели
- модельный клиент не знает Telegram
- orchestration находится в сервисах
- новые возможности добавляются через явные модули, а не через разрастание handlers
