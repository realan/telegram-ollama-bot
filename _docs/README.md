# Project Docs Workflow

Эта папка хранит не только документацию по проекту, но и рабочую систему постановки задач по фазам.

Цель:

- держать проектный контекст в компактном виде
- выполнять развитие проекта по фазам
- запускать каждую фазу в отдельном чате
- экономить токены и не таскать лишний контекст между итерациями

## Структура

- [_docs/vision.md](/Users/user/projects/hw1/_docs/vision.md) - краткое видение проекта и его смысл
- [_docs/context-dump.md](/Users/user/projects/hw1/_docs/context-dump.md) - актуальный снимок текущего состояния проекта
- [_docs/roadmap.yaml](/Users/user/projects/hw1/_docs/roadmap.yaml) - master-план развития по фазам
- [_docs/architecture/README.md](/Users/user/projects/hw1/_docs/architecture/README.md) - архитектурные документы
- [_docs/specs/README.md](/Users/user/projects/hw1/_docs/specs/README.md) - исполнимые phase spec для отдельных чатов
- [_docs/reports/README.md](/Users/user/projects/hw1/_docs/reports/README.md) - краткие отчеты по завершенным фазам
- `_docs/decisions/` - ADR и другие фиксированные архитектурные решения

## Роли документов

### `vision.md`

Отвечает на вопрос:

- зачем существует проект
- что в нем главное
- куда он в целом развивается

Это самый стабильный документ. Он меняется редко.

### `roadmap.yaml`

Отвечает на вопрос:

- какие фазы запланированы
- в каком порядке их делать
- какой у них статус
- от чего зависит каждая фаза

Это master-план.

### `context-dump.md`

Отвечает на вопрос:

- что сейчас правда о проекте
- что уже реализовано
- какие ограничения и риски есть сейчас
- какая следующая актуальная фаза

Это живой документ. Его нужно обновлять после завершения каждой фазы.

### `architecture/*`

Отвечает на вопрос:

- как сейчас устроен проект
- где границы модулей
- как идут runtime-сценарии
- по каким правилам развивать архитектуру

Это опорная документация для принятия решений и постановки фаз.

### `specs/phase-*.yaml`

Отвечают на вопрос:

- что именно делать в конкретной фазе
- что входит в scope, а что нет
- какие файлы можно менять
- какие критерии готовности и тесты нужны

Это исполнимые task-spec для отдельных чатов.

### `reports/*`

Отвечают на вопрос:

- что реально было сделано в фазе
- что осталось несделанным
- какие решения были приняты по ходу
- что важно для следующей фазы

Это краткая память о выполнении фазы, чтобы не потерять выводы из чата.

### `decisions/*`

Отвечают на вопрос:

- какие важные решения уже приняты
- почему выбрано именно это решение
- какие альтернативы были

Сюда стоит складывать ADR, если решение важно не потерять.

## Рабочий процесс

Нормальный цикл работы выглядит так:

1. Зафиксировать общее направление в `vision.md`.
2. Разбить развитие проекта на фазы в `roadmap.yaml`.
3. Для нужной фазы подготовить отдельную spec в `specs/phase-*.yaml`.
4. Запустить новый чат, передав:
   - `context-dump.md`
   - нужные architecture docs
   - конкретную `phase-*.yaml`
5. Выполнить фазу.
6. После завершения обновить:
   - статус в `roadmap.yaml`
   - `context-dump.md`
   - phase report в `reports/`
   - при необходимости `architecture/*`
   - при необходимости `decisions/*`

## Что давать в новый чат

Минимальный пакет:

- [_docs/context-dump.md](/Users/user/projects/hw1/_docs/context-dump.md)
- соответствующий phase spec из [_docs/specs/](/Users/user/projects/hw1/_docs/specs)

Расширенный пакет, если задача нетривиальная:

- [_docs/context-dump.md](/Users/user/projects/hw1/_docs/context-dump.md)
- нужные файлы из [_docs/architecture/](/Users/user/projects/hw1/_docs/architecture)
- phase spec
- ADR из `_docs/decisions/`, если они уже есть и важны для фазы

## Как понять, что документация в порядке

Система документов считается здоровой, если:

- `roadmap.yaml` показывает реальный статус фаз
- `context-dump.md` описывает текущее состояние, а не старую историю проекта
- у каждой активной фазы есть отдельная исполнимая spec
- у завершенных фаз есть короткие отчеты в `reports/`
- architecture docs не противоречат фактическому коду
- важные решения не теряются в переписке, а фиксируются в `decisions/`

## Минимальный ритуал после каждой фазы

После завершения фазы нужно сделать пять вещей:

1. Обновить статус фазы и задач в [_docs/roadmap.yaml](/Users/user/projects/hw1/_docs/roadmap.yaml).
2. Обновить [_docs/context-dump.md](/Users/user/projects/hw1/_docs/context-dump.md) под новое текущее состояние.
3. Сохранить краткий отчет по фазе в [_docs/reports/](/Users/user/projects/hw1/_docs/reports).
4. Если изменились архитектурные границы, обновить соответствующие файлы в [_docs/architecture/](/Users/user/projects/hw1/_docs/architecture).
5. Если было принято важное решение, добавить ADR в `_docs/decisions/`.

## Практический принцип

`vision` отвечает за направление.  
`roadmap` отвечает за план.  
`phase spec` отвечает за конкретную задачу.  
`context-dump` отвечает за актуальное состояние.  
`reports` отвечают за краткую память о выполнении фаз.  
`architecture` отвечает за устройство системы.  
`decisions` отвечают за память о принятых решениях.
