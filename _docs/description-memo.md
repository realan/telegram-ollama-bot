# Memo: хранение истории, суммаризация и команды бота

## Хранение истории

История хранится **в памяти процесса** (`InMemoryConversationStore`) — персистентность между перезапусками отсутствует.

Структура данных: `defaultdict[int, list[Message]]` — словарь, где ключ это `user_id` Telegram, значение — список сообщений пользователя.

Тип `Message` — `TypedDict` с двумя полями:
- `role`: `"system" | "user" | "assistant" | "summary"`
- `content`: `str`

Роль `"summary"` — специальная: это не роль Ollama API, а внутренний маркер сжатого контекста. При отправке в модель сообщение с этой ролью преобразуется в `user`-сообщение с префиксом `"Summary of earlier conversation:\n"` ([ollama_client.py:102-109](../app/clients/ollama_client.py#L102-L109)).

**Операции хранилища:**
- `read(user_id)` — возвращает копию истории (защита от мутации)
- `append / append_many` — добавляет сообщения
- `replace_after_summary` — атомарно заменяет весь список: `[summary_message, ...tail_messages]`
- `clear` — удаляет историю пользователя

После каждого ответа в историю сохраняются два сообщения: `user` и `assistant` ([chat_service.py:52-58](../app/services/chat_service.py#L52-L58)).

## Суммаризация

Суммаризация запускается **перед каждым ответом**, если в истории накопилось достаточно сообщений.

### Условие запуска

`_split_history_for_summary` ([chat_service.py:103-122](../app/services/chat_service.py#L103-L122)):

1. Если первое сообщение в истории — `"summary"`, оно исключается из подсчёта триггера.
2. Подсчитываются остальные (не-summary) сообщения.
3. Если их количество **≤ `SUMMARY_TRIGGER_MESSAGES`** (дефолт: 5) — суммаризация не нужна.
4. Иначе история разбивается: всё кроме `keep_tail` последних сообщений идёт на сжатие, `keep_tail` остаётся хвостом.

`keep_tail = max(SUMMARY_TRIGGER_MESSAGES - 2, 1)` — т.е. при дефолте 5 хвост = 3 сообщения.

### Процесс суммаризации

1. Сообщения для сжатия (включая предыдущий summary, если есть) отправляются в Ollama с системным промптом из `summarization_prompt.txt`.
2. Промпт инструктирует модель: сохранять только факты, предпочтения, решения и незакрытые задачи; без нарратива и фраз-заглушек.
3. Ответ модели нормализуется (`_normalize_summary_text`): схлопываются пробелы, текст обрезается до `SUMMARY_MAX_CHARS` (дефолт: 800) символов с добавлением `"..."` если был обрезан.
4. Результат сохраняется как `Message(role="summary", content=...)`.
5. В хранилище вызывается `replace_after_summary`: вся история заменяется на `[summary_message, ...tail_messages]`.

### Ограничения контекста перед отправкой в модель

Помимо суммаризации, `_trim_history_for_model` применяет два среза прямо перед отправкой (не меняет хранилище):

- **По количеству сообщений** (`MAX_HISTORY_MESSAGES`, дефолт: 10): если в списке больше N сообщений, отсекаются старые, но summary-сообщение всегда сохраняется на первом месте.
- **По суммарному объёму символов** (`MAX_CONTEXT_CHARS`, дефолт: 4000): если суммарная длина контента превышает лимит (с учётом системного промпта), старые сообщения удаляются поочерёдно; summary удаляется последним.

### Параметры конфигурации

| Переменная окружения       | Дефолт | Назначение                                      |
|----------------------------|--------|-------------------------------------------------|
| `MAX_HISTORY_MESSAGES`     | 10     | Лимит сообщений в контексте модели              |
| `MAX_CONTEXT_CHARS`        | 4000   | Лимит символов в контексте модели               |
| `SUMMARY_TRIGGER_MESSAGES` | 5      | Порог non-summary сообщений для запуска сжатия  |
| `SUMMARY_MAX_CHARS`        | 800    | Максимальная длина текста summary               |
| `SUMMARIZATION_PROMPT_PATH`| `app/prompts/summarization_prompt.txt` | Путь к промпту суммаризации |

## Команды бота

### /start

Приветственное сообщение. Описывает базовое поведение бота: история ведётся, `/clear` сбрасывает её, `/help` показывает команды.

Handler: [start.py](../app/bot/handlers/start.py). Не зависит от хранилища, не имеет side-эффектов.

### /help

Выводит список доступных команд: `/start`, `/help`, `/clear` и описание текстового режима.

Handler: [help.py](../app/bot/handlers/help.py). Не зависит от хранилища, не имеет side-эффектов.

### /clear

Сбрасывает историю диалога текущего пользователя. После выполнения следующий запрос обрабатывается без предыдущего контекста.

Handler: [clear.py](../app/bot/handlers/clear.py).

**Как работает:**
1. Handler получает `user_id` из `message.from_user`.
2. Вызывает `conversation_store.clear(user_id)` — удаляет запись пользователя из `_messages_by_user`.
3. Отвечает: `"История диалога очищена. Начинаем сначала."`
4. Если у пользователя не было истории — команда всё равно выполняется без ошибки.

**Dependency injection:** `conversation_store` передаётся в handler напрямую через dispatcher aiogram (`dispatcher["conversation_store"] = conversation_store` в [main.py](../app/main.py)). ChatService при этом не задействован — операция не является бизнес-логикой.

**Порядок регистрации роутеров** в [router.py](../app/bot/router.py): `start → help → clear → chat`. Роутер `/clear` стоит перед `chat.router`, чтобы команда не перехватывалась общим text-handler'ом.
